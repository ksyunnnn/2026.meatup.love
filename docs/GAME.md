# Meat & Greet — 会場交流ポイントゲーム 実装仕様

meatup 2026（開催日 2026-07-25）の会場で遊ぶ交流ゲーム。参加者どうしが QR を
交換して「繋がり」を作り、ポイントを競う。会場のプロジェクターにソーシャル
グラフが育つ。想定人数は50人弱。

- **仕様の一次情報（設計時）**: Issue#11
- **実装**: PR#16（ブランチ `feat/game-datamodel`）
- **本書の位置づけ**: 実装された結果としての仕様。Issue#11 の記述と食い違う
  場合は本書が正。差分は「1. Issue#11 からの変更点」に列挙する。

---

## 0. 用語

| 語 | 意味 |
|---|---|
| **繋がり** | 参加者2人のあいだに1本だけ作れる関係。片側がもう片方の QR を読むと成立する |
| **SR** | 公表されたボーナス保持者。会場スクリーンで発光し、交換用チケットに印が出る |
| **SSR** | 非公表のボーナス保持者。一般参加者と見分けがつかない。繋がって初めて分かる |
| **ボーナス点** | SR / SSR ごとに個別設定する点数。その人と繋がった相手が得る |
| **Mate** | 各参加者の画面に並ぶ、繋がった相手の一覧 |
| **edition** | 開催回の識別子。全ドキュメント共通で `"2026"` |

SSR のほうが上位レアなのは、公表されていない＝狙って当てにいけないため。

## 2. 採点規則

得点は1種類しかない。会場スクリーンの順位・各自の手元の点数・表彰用に凍結
した順位は、すべて `src/lib/game.ts` の `finalScoreFrom` を通るので食い違わ
ない。

```
得点 = Σ（繋がった各相手について）
         相手が SR / SSR なら → その相手の bonusPoints
         そうでなければ       → 1
```

自分が SR / SSR かどうかは、自分の得点に影響しない。

| 組み合わせ | 一方の得点 | 他方の得点 |
|---|---|---|
| 通常 × 通常 | 1 | 1 |
| 通常 × SR/SSR（ボーナス5） | 5 | 1 |
| SR（ボーナス5） × SSR（ボーナス10） | 10 | 5 |

ボーナスは常時反映される。したがって、ある参加者の点が大きく跳ねれば
「その人はいま SSR と繋がった」と推測できる。これは受け入れた設計で、
理由と退けた代案は `firestore.rules` の `specials` ブロックに記録がある。

## 3. データモデル

型定義は `src/lib/types.ts`。アクセス制御は `firestore.rules`。

| コレクション | ドキュメントID | 主なフィールド | 読み取り | 書き込み |
|---|---|---|---|---|
| `connections` | `{小さいuid}__{大きいuid}` | `a` `b` `by` `createdAt` | 全員 | サインイン済み（作成のみ） |
| `specials` | 参加者の uid | `bonusPoints` `public` `name` | サインイン済み（get・list とも） | 運営 |
| `control` | `2026` | `game` `ranking` `reveal` | 全員 | 運営 |
| `results` | `2026` | `top`（凍結した上位10名） | 全員 | 運営 |
| `staff` | `2026` | `uids`（SR のみ） | 全員 | 運営 |
| `admins` | 運営の uid | （空。存在することが権限を意味する） | サインイン済み（単一get） | **不可**。コンソール等の特権操作のみ |

### 重複防止

ドキュメントIDが2つの uid をソートして結合したものなので、同じペアの繋がりは
物理的に1件しか作れない。更新経路も持たせていない（`src/lib/game.ts` の
`connectionId`）。

### SSR が SR と区別される唯一の点

`staff/2026` に載らないため会場スクリーンで発光せず、交換用チケットにも印が
出ない。ボーナス点そのものは秘匿されない。

## 4. 画面

| パス | 対象 | 認証 | 機能 |
|---|---|---|---|
| `/game` | 参加者 | 参加確定者 | 交換用チケット提示／QRスキャン／下4桁手入力／自分の点数／Mate 一覧 |
| `/live` | 会場スクリーン | サインイン済み（運営でなくてよい） | ソーシャルグラフ／ポイント上位8名／結果発表演出 |
| `/control` | 司会 | 運営 | 開始・締め・ランキング伏せ・結果発表 |
| `/admin/specials` | 運営 | 運営 | SR / SSR とボーナス点の設定（選択した時点で保存） |

`/live` へは `/mypage` と `/game` からリンクしている。参加者も自分のスマホで
開ける。

### グラフのノードの大きさ

繋がりの本数で決まる（ポイントではない）。辺が1本しかないノードが巨大に描か
れると網の絵として破綻するため。

### 獲得演出（`/game`）

通常の繋がりは2.4秒で自動的に閉じる。SR / SSR は閉じる操作をするまで残り、
その間スキャンを停止する。前者は一晩に数十回起きるので次のスキャンを妨げて
はならず、後者は数回しか起きず端末を相手に向けて見せる類のものだから。

### 会場スクリーンのキー操作

司会リモート（`/control`）が不通のときの代替。`/live` にフォーカスした状態で
押す。運営としてサインインしている場合のみ有効。

| キー | 動作 |
|---|---|
| `L` | ランキングの伏せ／表示を切り替え |
| `R` | 結果発表を再生 |
| `Esc` | 結果発表を閉じる |

## 5. 当日の運営手順（2026-07-25）

1. **開場前** — `/admin/specials` で SR / SSR とボーナス点を設定する。
   SR は会場スクリーンで光るので、誰が高得点かを全員に知らせたい人に使う。
   SSR は一般参加者に混ぜる当たり。
2. **プロジェクター** — 運営アカウントでサインインした端末で `/live` を開く。
   `/control` を同じ端末で開いておくとキー操作の代替が効く。
3. **交流タイム中** — `/control` は「開催中」のまま。参加者は `/game` で交換
   する。
4. **終盤** — `/control` でランキングを伏せる（グラフは育ち続ける）。
5. **17:00 頃** — `/control` で締める（2段階確認）。締めるとスキャンが止まり、
   順位が凍結される。誤って締めた場合は「開催に戻す」で戻せる。
6. **表彰** — `/control` で結果を出す。演出をやり直したい場合は「もう一度」。
   閉じたい場合は「発表を閉じる」。

## 6. 本番反映の手順

`docs/DEPLOY.md` 準拠。Cloudflare Pages の git 自動デプロイは未設定なので手動。

1. `.env.local` を本番値にする
   （`NEXT_PUBLIC_FIREBASE_PROJECT_ID=meatup-2026` / `NEXT_PUBLIC_USE_EMULATOR=false`）
2. `npx firebase deploy --only firestore:rules --project meatup-2026`
   — **これを先に実行しないと全機能が permission-denied になる**
3. `npm run build`
4. `npx wrangler pages deploy out --project-name meatup-2026 --branch main`
